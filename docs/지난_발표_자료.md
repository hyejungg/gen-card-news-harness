## 1. 왜 요즘 agent 얘기는 모델 얘기만으로 설명이 안 되는가

모델 성능은 기본 전제임

**2023년: 도구 사용과 워크플로우 실험이 본격화**

- 모델이 외부 기능을 호출할 수 있게 된 단계 (모델의 능력 확장에 초점)
- agent 운영 구조가 아닌 모델이 도구를 불러서 더 많은 context 를 활용할 수 있게 되었다 느낌
- 단독으로 문제 해결보다는 여러 단계를 거쳐 작동하는 workflow 중심

**2024년: 프롬프트보다 orchestration과 구조 선택의 중요성이 부각**

> *좋은 프롬프트와 적절한 도구만으로 충분할 때도 있지만, 플로우가 많아지면 어려워진다. 
([Orchestrating Agents: Routines and Handoffs](https://developers.openai.com/cookbook/examples/orchestrating_agents?utm_source=chatgpt.com) - OpenAI)*
> 
- 이 문제를 오케스트레이션 관점으로 풀어보기 시작함
    - 여러 역할 조합으로 에이전트들을 나눠서 분업하여 실행되도록 함
- 관련하여 Anthropic에서 [에이전트를 효과적으로 만드는 방법](https://www.anthropic.com/engineering/building-effective-agents?utm_source=chatgpt.com)에 대한 아티클도 등장
    - 다양한 설계 패턴(체이닝, 라우팅, 오케스트레이션 등)에 대해 정리한 것으로 **workflow와 agent를 구분하고, 어떤 문제에 어떤 구조가 맞는지 선택하는 engineering 관점이 중심**

**2025년: agent를 제품/운영 시스템으로 다루는 도구와 SDK가 늘어남**

- OpenAI와 Anthropic 모두 좋은 프롬프트를 찾는 문제에서 상태 관리, 도구 조합, 관측 가능성, 평가 루프를 설계하는 문제로 관점이 이동함
    - OpenAI는 response api, built-in tools, agents sdk 등을 발표
    - Anthropic은 agents sdk, multi-agent collaboration, parallel agents 에 대해 설명
- 좋은 프롬프트를 짜는 법 → 상태 관리, 도구 조합, 평가 루프 설계하는 일로 변화

**2026년: 인간의 역할이 ‘직접 수행’보다 ‘환경·규칙·피드백 구조 설계’로 이동**

> “Humans steer. Agents execute.”
인간이 조종한다. 에이전트들이 실행한다.
> 
- **인간 엔지니어의 역할은 환경을 설계하고, 의도를 명세하고, 피드백 루프를 구축하는 일** (OpenAI-[Harness engineering: leveraging Codex in an agent-first world](https://openai.com/ko-KR/index/harness-engineering/?utm_source=chatgpt.com))
    - 에이전트가 일을 잘하게 하려면, 필요한 지식이 레포 안에 있고, 기계가 읽기 좋은 형태로 정리되어 있어야 함
    - 에이전트의 지침(AGENTS.md)를 길게 주지 말고, 레포 안에서 탐색 가능한 knowledge base 를 줄 것!
        - e.g
            - `AGENTS.md`: 이 레포에서 작업하는 기본 규칙
            - `docs/architecture/index.md`: 전체 구조 요약
            - `docs/services/payment.md`: 결제 서비스 설명
            - `plans/active/...`: 지금 진행 중인 작업
            - `debt/...`: 알려진 문제점
        - running in context에서 접근 할 수 없는 것은 agent 관점에선 존재하지 않는다!
- **작업이 여러 context window와 장시간 실행에 걸칠수록 더 효과적인 harness가 필요** (Anthropic-[Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents?utm_source=chatgpt.com))
    - claude agent sdk는 claude code를 구동하는 same core tools, context management systems, permission frameworks를 제공
        
        → 모델 / 도구 계층 / 컨텍스트 관리 계측 / 권한 계층으로 분리되어 있다는 의미
        
    - sandboxing 을 통해서 bash tool의 실행 제한을 두어 → 제한된 공간 안에서는 매번 승인을 받지 않도록 하여 permission prompts 를 줄임 (생산성 높임)
    - subagent는 자기 자신의 context window, tool access, system prompt 를 가짐 → 같은 모델을 쓰더라도 어떤 작업을 어떤 하위 agent에 위임하냐에 따라 결과가 달라짐

## 2. Harness란 정확히 무엇인가

- agent 가 신뢰 가능하게 일하도록 만드는 모델과는 별개의 운영 계층
    - 모델은 “다음에 무엇을 할지”를 제안한다면, 하네스는 “그 행동이 실제로 어떻게 실행되고 통제될지”를 정함
    - e.g
        - 사용자 요청을 작업 단위로 변경
        - 세션/스레드 상태를 유지
        - 도구 연결
        - 실행 권한 승인과 관련된 통제
        - 로그/관측/실패 복구 체계 등
- **MCP**: 외부 시스템 연결 표준
    - e.g. 배달앱
- **Skills**: 재사용 가능한 작업 지식/스크립트 패키지
    - e.g. 메뉴별 조리법 + 자주 쓰는 작업 메뉴얼
- **Harness**: 위 셋을 포함해 상태·권한·승인·복구·로그까지 묶어 실제 agent를 운영하는 계층
    - e.g. 가게가 잘 굴러가도록 만드는 운영 질서

## 3. Claude Code cli와 Codex cli는 무엇을 다르게 최적화하고 있나

> 요즘 핫한 claude code. 하지만 나는 codex cli를 쓰지. 
같은 코딩 에이전트지만, 어떤 점이 다를까?
> 

- 자율성, 통제성, 실행 방식에 대한 제품 철학 차이가 있음
    - Claude Code CLI: **“agent를 내 작업 흐름에 끼워 넣는 쪽”**
    - Codex CLI: **“내 작업 흐름을 agent runtime 위에 정렬하는 쪽”**

| 항목 | Claude Code CLI | Codex CLI |
| --- | --- | --- |
| **핵심 차이** | 확장점 중심 | 설정/정책 중심 |
| **하네스 관리** | workflow customization | runtime governance |
| 철학 | 사용자에게 최대한 물어보고 align을 맞추자 | 내가 다 알아서 해줄게  |
- claude code cli, codex cli 등 각각 하네스 자체를 사용자가 수정할 순 없지만 커스터마이징은 가능함
    - cladue code의 경우 plugins을 통해 진행
        - workflow 확장 중심으로 hooks, slash commands, agents, plugins, MCP 연결, 권한 설정
    - codex의 경우 설정 파일과 AGENTS.md 등으로 진행
        - config.toml, project override, AGENTS.md, sandbox/network/approval, MCP 등록

## 4. MCP vs CLI 논쟁의 진짜 쟁점은 무엇인가

### [MCP는 죽었다. CLI 만세](https://news.hada.io/topic?id=27129)

- 로컬에서 사용하는 에이전트 관점에서 CLI가 더 단순하고 디버깅이 가능하며 jq, grep 등과 같은 cli 명령어를 이미 llm이 잘 다룬다.
- 그에 비해 mcp는 별도 프로세스, 재인증, 거친 권한 제어 등으로 인해 생산성이 떨어진다.

### [MCP는 죽었다. MCP 만세](https://news.hada.io/topic?id=27530)

- 로컬 stdio mcp와 원격 streamable mcp를 사람들이 혼동한다.
- 로컬에서 사용하는 에이전트의 경우 cli가 유리할 수 있지만, 엔터프라이즈에서는 중앙 mcp 서버를 통해 인증 체계를 관리하고, observability하는게 중요하다.

### [Anthropic, mcp를 이용한 코드 실행: 더 효율적인 에이전트 구축](https://www.anthropic.com/engineering/code-execution-with-mcp?utm_source=chatgpt.com)

- 2024년 11월 mcp 출시 이후 엄청 많은 Mcp 서버가 구축됨
- 그러나 에이전트는 수백, 수천개로 도구 수가 증가함에 따라 도구 정의를 불러오고, 중간 결과를 context window 를 통해 전달하면서 에이전트가 느려지고 비용이 증가하는 문제가 있음
- 에이전트가 cli를 사용하는 것이 보편화함에 따라 mcp서버를 직접 도구 호출에만 쓰는 것이 아니라, 코드 api처럼 노출해서 모델이 코드를 작성해서 api를 쓸 수 있게 제안
    - 기존 mcp - 모델이 도구 하나하나 직접 호출하고 결과를 확인
    - code execution with mcp - 모델이 짧은 코드를 직접 작성해서 제시된 도구들을 코드 내에서 실행
    - e.g. 엑셀에서 1만 건 중 미결제 주문만 찾아줘
        - 기존 mcp
            - 스프레드시트 mcp 도구 호출 → 1만줄 데이터 → 모델이 읽어서 미결제 주문 찾음 → 모델이 결과를 정리해서 답변
        - code execution with mcp
            - python 코드로 스프레드 시트를 읽어서 미결제 주문을 찾아서 return → 모델이 결과를 정리해서 답변
- 코드 실행을 통해 에이전트가 mcp 서버를 효율적으로 사용하는 방법을 제시함

## 5. OpenAI와 Anthropic 사례로 보는 Agent Engineering의 핵심 설계 포인트

- 에이전트 시스템의 핵심은 모델 자체보다
    - 대화 상태를 어떻게 유지하고
    - 어떤 실행 환경에서
    - 어떤 도구를 통해
    - 어떤 통제 장치와 함께 일하게 할 것인가임
- 모델을 고르는 것보다 먼저, 작업 단위·실행 권한·도구 분업·복구 방식·관측 지표를 정해야 운영 가능한 Agent 시스템이 됨
    
    **1) 무엇을 하나의 작업 단위로 볼 것인가?**
    
    - 어떤 대화를 같은 thread로 볼 것인가
    - 어떤 작업을 같은 workspace로 볼 것인가
    - 어떤 memory를 다음 작업에도 넘길 것인가
    - 어떤 산출물과 로그를 같은 실행 단위로 묶을 것인가
    
    **2) 에이전트는 어디서, 어떤 권한으로 일할 것인가?**
    
    - 로컬 머신에서 돌 것인가, 원격 샌드박스에서 돌 것인가
    - 어떤 디렉터리를 읽고 쓸 수 있는가
    - 네트워크 접근은 가능한가
    - 어떤 명령은 자동 실행이고 어떤 명령은 승인 대상인가
    
    **3) 모델과 실행환경은 무엇을 나눠 맡을 것인가?**
    
    - 모델이 직접 모든 도구 호출을 해석하고 중개할 것인가
    - CLI를 통해 외부 도구를 다루게 할 것인가
    - code execution with MCP처럼 실행환경이 중간 처리를 맡게 할 것인가
    
    **4) 실패했을 때 어떻게 멈추고, 되돌리고, 다시 시작할 것인가?**
    
    - 어떤 행동은 자동 승인인가
    - 어떤 행동은 사람 확인이 필요한가
    - 실패하면 어디로 롤백하는가
    - 재시도는 같은 컨텍스트에서 할 것인가, 새 컨텍스트로 분기할 것인가
    - diff는 어디까지 보여줄 것인가
    
    **5) 무엇을 측정하고, 어떻게 비용을 통제할 것인가?**
    
    - 어떤 로그와 이벤트를 남길 것인가
    - latency, failure rate, approval rate, retry rate를 어떻게 볼 것인가
    - tool call 수와 token overhead를 어떻게 추적할 것인가
    - 어떤 단계가 병목인지 어떻게 식별할 것인가
    - 어떤 실패가 모델 문제인지, 어떤 실패가 구조 문제인지 어떻게 구분할 것인가

## 6. 결론: 2026년 agent 경쟁력은 어디서 갈리는가

- 지금은 AI 과도기에 탑승해 있다는 느낌을 다시 한 번 받음
    - 불과 얼마 전까지만 해도 모델 성능 경쟁이 중심이었는데,
     이제는 mcp / cli / 운영 등과 agent 운영 쪽의 논의로 이동한 느낌
- **결국 경쟁력은 좋은 모델 하나가 아니라 모델이 일하는 구조를 얼마나 잘 설계했는가에 달려있는 느낌**
